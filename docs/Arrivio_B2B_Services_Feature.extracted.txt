1. Overview
Add a Services feature to the Arrivio B2B Partner Platform. This allows HR admins and company
administrators to browse, request, and track relocation services for their employees- either
individually or in bulk.
The feature is inspired by the existing B2C services ﬂow (see Section 4- Visual Reference) but
adapted speciﬁcally for B2B use cases: multi-employee ordering, per-employee customization,
and centralized order tracking.
Entry points:
A Services strip on the main dashboard (quick discovery)
A Services nav item in the sidebar (primary navigation)

3. Before You Write Any Code
This is the most important section. Read it fully before doing anything else.
This feature is being added to an existing B2B codebase. Before writing a single line of code,
Codex must:
3.1 Read and Understand the Codebase
Explore the full project structure- understand how folders are organised, where pages
live, where components live, where API routes are deﬁned
Read at least 2–3 existing page components to understand the coding style, how state is
managed, how Axios is used, how props are structured
Read the existing sidebar/navigation component- understand exactly how nav items
are added so the new Services item follows the same pattern precisely
Read the existing dashboard page- understand the layout structure so the Services strip
is inserted correctly without breaking existing sections
Read existing modal components if any exist- follow the same pattern for the Request
Modal

===== PAGE 3 =====

Read existing backend route ﬁles- follow the same structure, middleware usage, error
handling, and response format already established
Read existing Supabase schema or migration ﬁles- understand existing table
structures, naming conventions, and foreign key patterns before creating new tables
3.2 Follow Existing Conventions- Always
File naming: match whatever convention is already used (PascalCase, camelCase, kebab-
case)
Folder structure: place new ﬁles exactly where similar existing ﬁles are placed
Component patterns: if existing components use a certain prop pattern, hooks pattern, or
state management approach- replicate it
API patterns: match the existing route structure, middleware chain, and response format
Styling: use the same Tailwind class patterns already used in the codebase- do not
introduce new design tokens or override existing ones
Database: follow the existing table naming conventions, column naming style, and foreign
key references
3.3 Do Not Assume
Do not assume folder paths- read the codebase ﬁrst
Do not assume table names- check existing Supabase schema ﬁrst
Do not assume how auth/session works- read existing authenticated routes ﬁrst
Do not assume how company_id or employee_id is stored- read existing data models
ﬁrst

POST /api/orders- Key Request Fields
The order creation endpoint must handle:
service_id- which service is being ordered
recipient_type- "all" or "specific"
employee_ids- array of employee IDs (used when recipient_type is "specific")
quantity- number per employee
special_requests- optional free text
custom_fields- object keyed by employee ID, containing per-employee customization
data
When recipient_type is "all", the backend should fetch all active employees under the
company and create an order row for each one (or a single order with a null employee_id-
decide based on what makes the My Orders view easier to query).
GET /api/orders- Supported Filters
status- ﬁlter by order status
employee_id- ﬁlter by speciﬁc employee
Pagination (follow existing pagination pattern in the codebase)
